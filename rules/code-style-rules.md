# 代码风格与协作规范

## 1. 语言规范
- 默认使用中文进行交流和文档编写
- 控制台错误和警告信息必须使用英文
- 代码注释使用中文

## 2. 需求评审流程
- 对需求进行专业评估，包括：
  - 需求合理性分析
  - 技术可行性评估
  - 潜在风险提示
  - 替代方案建议
- 未经明确确认（【开始修改】或【继续修改】指令）不进行代码修改

## 3. 代码注释规范

### 3.1 方法注释
```typescript
/**
 * 方法用途的简要说明
 * 
 * @param {类型} 参数名 - 参数说明
 * @param {类型} [可选参数名] - 可选参数说明（默认值：xxx）
 * @returns {返回类型} 返回值说明
 * @throws {异常类型} 可能抛出的异常及条件
 * @example
 * // 使用示例
 * functionName(param1, param2);
 */
```

### 3.2 类注释
```typescript
/**
 * 类的详细说明，包括主要功能和用途
 * 
 * @property {类型} 属性名 - 属性说明（默认值：xxx）
 * @event 事件名 - 事件说明
 * @example
 * // 使用示例
 * const instance = new ClassName();
 */
```

### 3.3 接口注释
```typescript
/**
 * 接口的详细说明
 * 
 * @property {类型} 属性名 - 属性说明（可选/必填）
 * @method 方法名 - 方法说明
 * @event 事件名 - 事件说明
 * @example
 * // 使用示例
 * const obj: InterfaceName = { ... };
 */
```

### 3.4 属性注释
```typescript
/** 属性用途说明（类型，默认值：xxx） */
private propertyName: Type = defaultValue;
```

### 3.5 核心逻辑注释
- 在复杂逻辑块前添加注释说明算法思路
- 非常规写法需要说明原因
- 临时解决方案需要标注 TODO 和待优化点

## 4. 错误处理规范
- 所有错误信息必须使用英文
- 错误信息应当清晰、具体，包含足够上下文
- 使用适当的错误级别（error/warn/info/debug）

## 5. 版本控制规范
- 提交信息使用中文
- 遵循约定式提交规范（Conventional Commits）
- 关联 issue 编号（如：fix #123）

## 6. 文档规范
- 项目文档使用中文编写
- 包含完整的 README.md
- 重要的设计决策记录在 DECISIONS.md 中

## 7. 代码审查
- 检查是否符合本规范
- 关注代码可读性和可维护性
- 确保注释与代码同步更新
